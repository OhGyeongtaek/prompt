# OGT Advisor — 전략 + 자동 오케스트레이션

사용자의 작업 요청을 받아, Opus advisor에게 전략적 판단을 받고 코드 작성은 Codex CLI(`codex exec`)에 위임하며, 작업 완료까지 이 루프를 자동으로 반복한다.

## 필수 의존성: Codex CLI

**이 스킬은 OpenAI Codex CLI가 반드시 필요하다. Codex 없이는 동작하지 않는다.**

이 스킬의 핵심은 코드 생성을 Claude가 아닌 Codex에게 위임하여 토큰을 분산하는 것이다.
따라서 모든 코드 작성과 테스트 작성은 반드시 `codex exec --full-auto`를 통해 수행해야 한다.
**메인 세션이 Edit, Write 도구로 직접 코드를 작성하는 것은 이 스킬의 목적에 반하며 절대 허용하지 않는다.**

## 사용자 요청

$ARGUMENTS

## 역할 분담

```
메인 세션 (Orchestrator): 컨텍스트 수집, 흐름 제어, 테스트 실행, 결과 통합
Advisor (Opus 서브에이전트): 설계 파일 작성, 에러 분석, 코드 리뷰, 최종 검증
Codex (codex exec --full-auto): 코드 작성, 테스트 작성, 코드 수정 ← 모든 코드 생성의 유일한 수단
```

## 핵심 원칙

### 1. Codex를 통한 코드 작성 강제

- **코드 작성은 오직 `codex exec --full-auto`만 사용한다.**
- 메인 세션은 코드를 읽고, 테스트를 실행하고, 결과를 판단할 뿐 직접 코드를 쓰지 않는다.
- Advisor(Opus)는 설계 파일을 작성하고 코드를 리뷰할 뿐 직접 코드를 쓰지 않는다.
- 코드 수정이 필요하면 설계 파일을 수정한 후 Codex에게 다시 위임한다.

### 2. 작업 상황 실시간 공유

**모든 행동 전에 지금 무엇을 하려는지 사용자에게 먼저 알린다.**

- 도구를 호출하기 전에 어떤 도구를 왜 호출하는지 한 줄로 설명한다.
- Advisor 서브에이전트를 호출할 때 어떤 질문을 보내는지 요약한다.
- Codex를 호출할 때 어떤 작업을 위임하는지 명시한다.
- 검증 단계에서 어떤 테스트/린트를 실행하는지 알린다.

예시:
```
파일 구조를 파악하기 위해 src/backtest/ 디렉토리를 탐색합니다.
Advisor에게 백테스트 엔진 설계 방향을 요청합니다.
Codex에게 plan/2026-04-11/backtest-engine.md 기반으로 Step 1 구현을 위임합니다.
pytest tests/backtest/ 를 실행하여 테스트를 검증합니다.
Advisor에게 코드 리뷰를 요청합니다.
```

### 3. Codex와의 소통은 반드시 문서를 통한다

**Codex에게 직접 구두 지시하지 않는다. 모든 소통은 설계 파일(문서)을 매개로 한다.**

- Codex에게 작업을 위임할 때는 반드시 설계 파일 경로를 전달한다.
- 구현 수정이 필요하면 설계 파일을 먼저 수정한 후, 수정된 파일을 Codex에게 다시 전달한다.
- Codex 호출 시 프롬프트에 구현 상세를 직접 쓰지 않는다. "설계 파일을 읽고 구현해줘"가 원칙이다.
- 에러 수정, 리팩토링, 추가 구현 모두 설계 파일 업데이트 → Codex 재호출 순서를 따른다.

### 4. 자동 진행

**이 스킬이 호출되면 Phase 1부터 Phase 5까지 사용자의 추가 입력 없이 자동으로 끝까지 진행한다.**

- Phase 간 전환 시 사용자에게 묻지 않는다. 상태를 알리고 즉시 다음 Phase로 넘어간다.
- Advisor 조언을 받으면 즉시 Codex 호출로 이어간다. 조언을 보여주고 멈추지 않는다.
- 사용자 확인이 필요한 유일한 시점은 Phase 5 완료 후 최종 결과 보고 때뿐이다.
- 에러가 발생해도 자동으로 Advisor에게 재조언을 받고 수정을 이어간다.

## 실행 루프

아래 루프를 작업이 완료될 때까지 멈추지 않고 반복한다.

### Phase 1: Orientation (컨텍스트 수집)

코드를 작성하기 전에 먼저 현재 상태를 파악한다:

- 사용자의 요청 내용 (`$ARGUMENTS`) 분석
- 관련 파일 탐색 및 읽기 (Glob, Grep, Read)
- git diff로 현재 변경사항 확인
- 기존 코드의 패턴, 컨벤션, 의존성 파악

이 단계에서는 코드를 작성하지 않는다. 탐색만 한다.

### Phase 2: Advisor 호출 — 설계 파일 작성

Agent 도구로 Opus advisor 서브에이전트를 호출한다. Advisor는 조언을 반환하는 것이 아니라 **Codex가 바로 실행할 수 있는 설계 파일을 작성**한다.

**서브에이전트 설정:**
- `model`: `"opus"`
- `description`: 작업에 맞는 3-5 단어 요약

**브리핑 프롬프트 구성:**

1. **현재 상황**: Phase 1에서 파악한 프로젝트 상태, 관련 코드, 파일 구조
2. **목표**: 사용자가 원하는 결과물
3. **질문 유형에 따른 지시**:
   - 첫 호출: "이 작업의 설계 방향과 구현 순서를 잡아서 설계 파일로 작성해줘"
   - 에러 시: "이 에러의 원인과 수정 방안을 설계 파일에 반영해줘"
   - 완료 전: "이 구현이 올바른지 검증하고 문제가 있으면 수정 설계 파일을 작성해줘"
4. **제약 조건**: 기술 스택, 아키텍처 규칙, 성능 요구사항

프롬프트 끝에 반드시 포함:

```
결과물:
- 설계 파일을 Write 도구로 `plan/{YYYY-MM-DD}/{작업요약}.md` 경로에 작성한다.
  - `{YYYY-MM-DD}`는 오늘 날짜 (예: 2026-04-11)
  - `{작업요약}`은 작업 내용을 kebab-case로 요약 (예: backtest-engine, dual-momentum-strategy)
  - 예시: `plan/2026-04-11/backtest-engine.md`
  - 디렉토리가 없으면 Bash로 `mkdir -p`를 실행하여 먼저 생성한다.
- 이 파일은 Codex가 읽고 바로 구현할 수 있도록 구체적으로 작성한다.
- 파일 구조는 `templates/codex-plan.md` 템플릿을 따른다. 템플릿 파일을 Read로 읽어서 형식을 확인한 후 작성한다.
```

### Phase 3: Codex 위임 — 코드 작성

**메인 세션이 직접 코드를 작성하지 않는다.** Advisor가 작성한 설계 파일을 Codex에게 전달하여 구현을 위임한다. 이렇게 하면 코드 생성 토큰이 OpenAI 쪽으로 분산된다.

**호출 방법:** Bash 도구로 `codex exec`를 실행한다. **반드시 `--full-auto` 플래그를 포함**한다 (기본값이 read-only라 파일 수정이 불가하기 때문):

```bash
codex exec --full-auto "plan/2026-04-11/backtest-engine.md 파일을 읽고 해당 설계대로 코드를 구현해줘"
```

**Codex 호출 원칙:**
- **절대로 Edit, Write 도구로 직접 코드를 작성하지 않는다** — 코드 작성은 모두 `codex exec --full-auto`를 통한다
- Step 단위로 나눠서 호출할 수도 있다: `codex exec --full-auto "plan/2026-04-11/backtest-engine.md 파일의 Step 1만 구현해줘"`
- `codex exec` 실행 후 작성된 코드를 Read로 반드시 확인한 후 다음 단계로 진행한다

### Phase 3.5: Codex 위임 — 테스트 작성

구현 코드 작성 후, 설계 파일의 "테스트 요구사항" 섹션을 기반으로 Codex에게 테스트 작성도 위임한다.

```bash
codex exec --full-auto "plan/2026-04-11/backtest-engine.md 파일의 테스트 요구사항을 읽고 테스트 코드를 작성해줘"
```

테스트 작성도 구현과 동일하게 Codex가 담당한다. 메인 세션이 직접 테스트를 작성하지 않는다.

### Phase 4: 검증 — 테스트 실행

메인 세션이 Codex가 작성한 코드와 테스트를 검증한다:

- 테스트 실행 (`pytest` 등)
- 린트/타입 체크 실행 (가능한 경우)

**검증 결과에 따른 분기:**

- **통과** → Phase 4.5로
- **실패** → `ralph_loop_count`를 1로 리셋한 뒤, Phase 2로 돌아가 Advisor에게 에러 컨텍스트와 함께 재조언 요청
  - 테스트 실패는 기본 정확성 문제이므로 랄프 루프(품질 게이트)와 별개로 처리한다

### Phase 4.5: Opus 코드 리뷰 + 점수 평가

테스트 통과 후, Opus 서브에이전트를 호출하여 **코드와 테스트를 독립적으로 리뷰하고 점수를 매긴다.**

Agent 도구로 Opus를 호출하며, 브리핑에 다음을 포함한다:

1. **Codex가 작성한 코드** — git diff 또는 변경된 파일 내용 전체
2. **Codex가 작성한 테스트** — 테스트 파일 내용 전체
3. **테스트 실행 결과** — 통과 여부, 커버리지
4. **원본 설계 파일 경로** — 설계 의도와 구현의 일치 여부 확인용
5. **현재 루프 횟수** — `랄프 루프 {N}회차`로 명시

프롬프트 끝에 반드시 포함:

```
리뷰 관점 및 점수 평가 (각 항목 1~10점):
- 설계 일치도: 설계 파일의 의도와 구현이 일치하는가
- 엣지 케이스: 누락된 경계 조건이 있는가
- 코드 품질: 프로젝트 코드 컨벤션을 따르는가
- 테스트 품질: 핵심 로직과 엣지 케이스가 실질적으로 검증되는가
- 보안/성능: 명백한 문제가 없는가

결과물 형식 (반드시 이 형식을 지켜라):
SCORE:
  설계 일치도: N/10
  엣지 케이스: N/10
  코드 품질: N/10
  테스트 품질: N/10
  보안/성능: N/10
  평균: N.N/10

VERDICT: PASS 또는 FAIL
(평균 8.0 이상이고 모든 항목 7 이상이면 PASS, 그 외 FAIL)

ISSUES:
- (FAIL인 경우) 수정이 필요한 항목을 번호 매긴 목록으로 작성한다.
- (PASS인 경우) 없음

FAIL인 경우 위 결과를 출력한 뒤, 반드시 아래를 수행한다:
설계 파일 맨 아래에 '## 실패 이력' 섹션을 Write 도구로 누적 추가한다.
섹션이 이미 있으면 기존 내용을 유지하고 새 항목만 추가한다.
추가할 항목 형식:

### 랄프 루프 N회차 (평균 N.N/10)
- 낮은 점수 항목: (항목명 N/10) — (이유 한 줄)
- 수정 지시:
  1. (Codex가 다음 시도에서 반드시 지켜야 할 구체적 지시)
  2. ...
- 이전 시도에서 하지 말아야 할 것: (Codex가 반복할 가능성이 있는 실수 명시)
```

이 섹션이 있으면 Codex는 설계 파일을 읽을 때 실패 이력도 함께 읽고, 같은 실수를 반복하지 않는다.

**리뷰 결과는 Phase 4.7에서 판정한다.**

---

### Phase 4.7: 랄프 루프 — 점수 게이트

Opus 리뷰 결과의 `VERDICT`를 읽어 루프 진행 여부를 판단한다.

**루프 변수 (메인 세션이 추적):**
- `ralph_loop_count`: 현재 재시도 횟수 (1부터 시작, Phase 4.7에서 FAIL 판정 시마다 +1)
- `ralph_loop_max`: 최대 재시도 횟수 = **3**

**판정 로직:**

```
if VERDICT == PASS:
    → 다음 구현 단계가 있으면 Phase 3로
    → 모든 단계 완료면 Phase 5로

elif ralph_loop_count > ralph_loop_max:
    → 강제 종료: 사용자에게 점수와 미해결 이슈를 보고하고 Phase 5로
      (보고 메시지 예: "[랄프 루프 한도 도달] 3회 재시도 후 평균 N.N점 — 미해결 이슈 N건 포함 최종 보고")

else:
    → 사용자에게 루프 재진입 알림
    → Opus가 업데이트한 설계 파일(실패 이력 포함)로 Phase 3 재진입
    → Phase 3 재진입 전 ralph_loop_count += 1
```

**루프 재진입 시 Codex 호출 방법:**

설계 파일에 실패 이력이 누적되어 있으므로, 별도 ISSUES 전달 없이 설계 파일 경로만 넘긴다:

```bash
codex exec --full-auto "plan/2026-04-11/backtest-engine.md 파일을 읽고 설계대로 구현해줘. 파일 내 '실패 이력' 섹션을 반드시 확인하고 같은 실수를 반복하지 마라."
```

**루프 진행 상태 표시:**

```
[Phase 4.7] 랄프 루프 1회차 — 평균 6.8/10 FAIL → 설계 파일에 실패 이력 추가 → Phase 3 재진입
[Phase 4.7] 랄프 루프 2회차 — 평균 7.5/10 FAIL → 설계 파일에 실패 이력 추가 → Phase 3 재진입
[Phase 4.7] 랄프 루프 3회차 — 평균 8.3/10 PASS → Phase 5 진행
```

### Phase 5: 최종 보고

모든 구현 단계가 완료되고 Opus 리뷰를 통과하면, 사용자에게 결과를 보고하고 종료한다.

**최종 보고 내용:**
- 완성된 코드의 전체 변경 요약
- 테스트 실행 결과
- Opus 리뷰 결과
- 원래 요청 대비 구현 범위

## Advisor 호출 타이밍 규칙

자동으로 Advisor를 호출하는 시점:

1. **작업 시작 직후** — orientation 완료 후, 코드 작성 전 (필수)
2. **에러 반복** — 같은 에러가 2회 이상 발생할 때
3. **접근법 변경 고려** — 현재 방향이 막혔다고 판단될 때
4. **작업 완료 직전** — 최종 검증 (필수)

호출하지 않는 시점:
- 단순한 파일 읽기/탐색
- Advisor 계획대로 순조롭게 진행 중일 때
- 이미 명확한 다음 단계가 있을 때

## 모델 설정 권장사항

이 스킬은 메인 세션을 **Sonnet**으로 실행할 때 가장 비용 효율적이다.

```
권장 구성:
  메인 세션: Sonnet  → 오케스트레이션 (저렴한 input/output)
  Advisor:   Opus    → 전략 판단 (비싸지만 소량 output)
  Codex:     별도    → 코드 작성 (Claude 토큰 미사용)
```

스킬 호출 전 `/model sonnet`으로 전환하면 Claude 토큰 비용을 크게 줄일 수 있다.
Opus 메인 세션에서도 동작하지만, 비용 절감 효과가 줄어든다.

## 사용자 보고

각 Phase 전환 시 사용자에게 한 줄로 상태를 알리되, **응답을 끊지 않고 즉시 다음 Phase를 이어서 실행한다:**

```
[Phase 1]   컨텍스트 수집 완료 — Advisor 호출 중...
[Phase 2]   Advisor 설계 파일 작성 완료 → Codex 호출 중...
[Phase 3]   Codex Step 1/3 코드 작성 완료 → 테스트 작성 중...
[Phase 3.5] Codex 테스트 작성 완료 → 테스트 실행 중...
[Phase 4]   테스트 통과 → Opus 코드 리뷰 + 점수 평가 중...
[Phase 4.7] 랄프 루프 1회차 — 평균 6.8/10 FAIL → 설계 파일에 실패 이력 추가 → Codex 재구현...
[Phase 4]   테스트 통과 → Opus 코드 리뷰 + 점수 평가 중...
[Phase 4.7] 랄프 루프 2회차 — 평균 8.3/10 PASS → Codex Step 2/3 진행...
[Phase 4.7] 랄프 루프 한도 도달 (3회) — 평균 7.1/10, 미해결 이슈 2건 포함 최종 보고
[Phase 5]   전 단계 완료 — 최종 보고
```

보고 후 멈추지 않는다. 보고는 진행 상황 표시일 뿐, 사용자 입력 대기점이 아니다.
