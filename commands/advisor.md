# Advisor — 독립적 전략 리뷰어

현재 작업에 대해 독립적인 시각의 전략적 조언을 제공하는 Opus advisor 서브에이전트를 호출한다.

## 사용자 요청

$ARGUMENTS

## 실행 지침

아래 단계를 순서대로 수행한다.

### 1단계: 컨텍스트 수집

advisor에게 전달할 브리핑을 구성하기 위해 현재 상태를 파악한다:

- 사용자의 요청 내용 (`$ARGUMENTS`)
- 현재 작업 중인 파일이나 코드가 있다면 해당 내용
- 최근 대화에서 논의된 결정사항이나 문제점
- git diff 등 변경 사항이 있다면 확인

### 2단계: Advisor 서브에이전트 호출

Agent 도구를 사용하여 Opus advisor를 호출한다. 반드시 아래 규칙을 따른다:

**서브에이전트 설정:**
- `model`: `"opus"` 고정
- `subagent_type`: 생략 (general-purpose)
- `description`: 3-5 단어로 advisor 역할 요약

**브리핑 프롬프트 작성 규칙:**

프롬프트에 반드시 포함할 내용:
1. **배경**: 프로젝트가 무엇이고 현재 어떤 작업을 하고 있는지
2. **질문**: 사용자가 advisor에게 묻고 싶은 것
3. **컨텍스트**: 관련 코드, 파일 경로, 에러 메시지, 이전 시도 등
4. **제약 조건**: 기술 스택, 아키텍처 제약, 성능 요구사항 등

프롬프트 끝에 반드시 다음 지시를 포함한다:

```
응답 규칙:
- 100단어 이내로 간결하게 답변한다
- 번호 매긴 액션 아이템으로 구성한다
- 설명이 아닌 판단과 방향을 제시한다
- 근거가 있다면 한 줄로 이유를 덧붙인다
- 리스크나 주의사항이 있으면 마지막에 명시한다
```

### 3단계: 결과 전달

서브에이전트의 응답을 사용자에게 전달할 때:

- advisor의 조언을 그대로 보여준다
- 조언에 대한 본인의 판단(동의/이견)이 있으면 간단히 덧붙인다
- 조언과 현재 코드/상태가 충돌하면 그 지점을 명시한다
- 다음 작업으로 바로 이어갈 수 있도록 구체적 액션을 제안한다

## 호출 타이밍 가이드

이 스킬은 다음 상황에서 가장 효과적이다:

1. **작업 착수 전** — 복잡한 구현의 방향 설정이 필요할 때
2. **막혔을 때** — 에러가 반복되거나 접근법이 수렴하지 않을 때
3. **선택지 비교** — 두 가지 이상 접근법 중 결정이 필요할 때
4. **완료 직전 검증** — 구현이 맞는 방향인지 최종 확인할 때

단순한 Q&A, 이미 명확한 다음 단계가 있는 경우에는 호출하지 않는다.
